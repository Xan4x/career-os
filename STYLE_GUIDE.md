# Style Guide

Ce document définit le style des livrables générés par Career OS.

Il ne décrit pas le fonctionnement du projet, uniquement la manière d'écrire les documents produits.

---

# Objectif

L'objectif n'est pas d'impressionner.

L'objectif est de communiquer une expérience professionnelle de manière :

- claire ;
- honnête ;
- crédible ;
- technique ;
- professionnelle.

Un bon document doit donner envie à un recruteur ou à un responsable technique de me rencontrer.

---

# Langues

## Base de connaissances

Français.

Toutes les informations du dépôt sont maintenues en français.

## Livrables

Anglais professionnel par défaut.

Sauf demande explicite, tous les documents générés doivent être écrits en anglais.

Ils doivent être rédigés comme s'ils avaient été écrits directement par un professionnel IT expérimenté.

Il ne faut jamais faire une traduction littérale du français.

---

# Style d'écriture

Privilégier un style :

- simple ;
- professionnel ;
- naturel ;
- précis ;
- orienté résultats.

Les phrases doivent être courtes.

Chaque phrase doit apporter une information utile.

---

# Priorités

Toujours mettre en avant :

1. les projets réalisés ;
2. les choix d'architecture ;
3. les réalisations ;
4. les résultats obtenus ;
5. les technologies utilisées.

Les technologies ne doivent jamais être présentées comme une simple liste.

Elles doivent être utilisées pour illustrer :

- une réalisation ;
- une compétence ;
- un choix d'architecture ;
- ou une connaissance pertinente pour le poste visé.

La valeur d'une technologie réside autant dans les concepts qu'elle démontre que dans son utilisation elle-même.

---

# Compétences

Les compétences ne sont jamais inventées.

La base de connaissances peut contenir :

- des expertises actuelles ;
- des expériences récentes ;
- des expériences historiques ;
- des connaissances techniques.

Toutes ces informations sont utiles et doivent être conservées.

En revanche, les livrables (CV, LinkedIn, lettres de motivation...) ne doivent présenter que les informations les plus pertinentes pour l'objectif recherché.

Une technologie historique peut parfaitement être conservée dans la base de connaissances sans apparaître dans un CV.

---

# Honnêteté

Ne jamais exagérer une responsabilité.

Ne jamais transformer une connaissance historique en expertise actuelle.

Toujours distinguer :

- l'expertise actuelle ;
- l'expérience récente ;
- les expériences historiques ;
- les connaissances techniques générales.

La base de connaissances conserve l'ensemble du parcours.

Les livrables sélectionnent uniquement les informations pertinentes pour leur objectif.

---

# CV

Le CV doit être :

- moderne ;
- sobre ;
- lisible ;
- orienté projets ;
- compatible ATS.

L'objectif est qu'un recruteur comprenne rapidement :

- qui je suis ;
- ce que je sais faire ;
- ce que j'ai réalisé.

---

# LinkedIn

Le profil LinkedIn peut être légèrement plus narratif que le CV.

Il doit cependant rester :

- professionnel ;
- technique ;
- crédible.

---

# Lettres de motivation

Les lettres de motivation doivent être personnalisées.

Elles doivent être générées à partir :

- de la base de connaissances ;
- de l'offre d'emploi.

Une lettre générique ne doit jamais être produite.

---

# Principe général

La qualité est plus importante que la quantité.

Il vaut mieux présenter cinq projets bien décrits que quinze technologies sans contexte.
